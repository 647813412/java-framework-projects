## 1. 项目概述

**优图（YouPicture）** 是一个面向个人与团队的图片管理与分享平台。核心诉求是让用户能够上传、管理、浏览图片，并支持私有空间、团队空间、公共图库三种场景。

平台目标：
- 支持用户个人图库（私有空间）管理。
- 支持团队协同图片管理（团队空间）。
- 提供公共图库供所有人浏览，图片上传前需管理员审核。
- 提供 AI 辅助能力（图片扩图、文生图、图片编辑、自动打标签）。
- 提供社交互动（评论、点赞）。
- 提供图片数据分析（空间使用情况、标签分类统计等）。

### 2.2 图片模块

| 编号 | 功能 | 描述 | 权限 |
|------|------|------|------|
| P-01 | 文件上传图片 | 上传本地文件到 COS，支持新增与覆盖更新 | 登录用户 |
| P-02 | URL 上传图片 | 通过图片 URL 抓取并上传到 COS | 登录用户（需空间上传权限） |
| P-03 | 编辑图片信息 | 修改名称、分类、标签、简介等元数据 | 登录用户（需空间编辑权限） |
| P-04 | 删除图片 | 删除图片记录并异步清理 COS 文件 | 登录用户（需空间删除权限） |
| P-05 | 查看图片详情 | 获取图片详情及当前用户操作权限列表 | 公开（私有空间需权限） |
| P-06 | 分页查询图片列表 | 公共图库/空间图库分页查询，支持多条件筛选 | 公开（空间图库需权限） |
| P-07 | 管理员审核图片 | 对公共图库上传的图片进行通过/拒绝审核 | 管理员 |
| P-11 | 颜色相似搜索 | 按主色调搜索相似图片 | 登录用户 |
| P-12 | AI 扩图 | 调用阿里云 AI 对图片进行扩图，异步轮询任务结果 | 登录用户（需空间编辑权限） |
| P-13 | AI 文生图 | 调用阿里云 AI 按文本描述生成图片 | 登录用户 |
| P-15 | AI 自动打标签 | 调用阿里云通义千问 VL 自动分析图片生成标签和分类 | 登录用户 |

**图片审核状态：**
- `0` 待审核（REVIEWING）
- `1` 通过（PASS）
- `2` 拒绝（REJECT）

**审核规则：**
- 私有空间上传：自动通过审核。
- 管理员上传公共图库：自动通过审核。
- 普通用户上传公共图库：进入待审核状态，需管理员审核。

**公开/私有字段（isPublic）：**
- 公共图库图片 `isPublic=1`，空间图片 `isPublic=0`。
- 用户个人主页仅显示本人的公开图片（或自己全部图片）。

### 2.3 空间模块

| 编号 | 功能 | 描述 | 权限 |
|------|------|------|------|
| S-01 | 创建空间 | 创建私有或团队空间，普通用户每类空间只能创建一个 | 登录用户 |
| S-02 | 删除空间 | 删除空间，团队空间同步清理成员记录 | 空间管理员 |
| S-03 | 更新空间 | 更新空间名称、级别等信息 | 空间管理员 |
| S-04 | 编辑空间 | 修改空间名称、头像 | 登录用户（空间创建者） |
| S-05 | 查看空间详情 | 获取空间信息及当前用户权限列表 | 登录用户 |
| S-06 | 分页查询空间列表 | 多条件筛选 + 分页 | 登录用户 |
| S-07 | 空间级别列表 | 获取所有空间级别的配额信息 | 公开 |

**空间类型：**
- `0` 私有空间（PRIVATE）
- `1` 团队空间（TEAM）

**空间级别及配额：**

| 级别 | 值 | 最大图片数 | 最大总容量 |
|------|----|-----------|-----------|
| 普通版 | 0 | 100 张 | 100 MB |
| 专业版 | 1 | 1000 张 | 1000 MB |
| 旗舰版 | 2 | 10000 张 | 10000 MB |

> 普通用户只能创建普通版空间，创建更高级别空间需要管理员权限。

**额度管理：** 上传图片时事务性增加 `totalSize` 和 `totalCount`；删除图片时事务性减少额度。

## 3. 非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 图片列表分页查询支持 Caffeine 本地缓存 + Redis 双级缓存（已实现，标注 @Deprecated 待迁移）；搜索历史异步记录，不影响主链路性能 |
| 安全 | 密码 MD5 + 盐值加密存储；接口基于 Sa-Token + 自定义 AOP（`@AuthCheck`）进行角色校验；私有空间基于 Sa-Token 扩展 Space 鉴权域进行权限细粒度控制 |
| 可靠性 | 图片上传与空间额度更新在同一事务中完成；图片删除与额度释放事务性执行；COS 文件清理异步执行，不阻塞主流程 |
| 扩展性 | 空间图片支持 ShardingSphere 按 spaceId 动态分表（`picture_{spaceId}`），公共图库路由到主表 `picture` |
| 可维护性 | 接口文档通过 Knife4j（OpenAPI 2）自动生成，访问 `/doc.html` |
| 防爬虫 | 列表查询每页最大 20 条，超过则报 PARAMS_ERROR |
| 文件安全 | 上传文件大小最大 10MB；URL 上传时校验协议（http/https）和 URL 格式 |
| 会话 | Session 基于 Redis 存储，Cookie 有效期 30 天；Sa-Token 整合 Redis Jackson 序列化 |

### 5.3 权限与鉴权方案

#### 5.3.1 全局角色权限（`@AuthCheck`）

基于自定义注解 `@AuthCheck(mustRole = "admin")` + AOP 切面 `AuthInterceptor` 实现。

```
请求 → AuthInterceptor.doBefore()
  → 获取 Session 中 loginUser
  → 校验 mustRole，不满足则抛 NO_AUTH_ERROR
```

#### 5.3.2 空间细粒度权限（Sa-Token 扩展鉴权域）

通过扩展 Sa-Token 自定义 `StpKit.SPACE` 鉴权域，实现 `@SaSpaceCheckPermission` 注解。

**权限来源：** 从 `spaceUserAuthConfig.json` 配置文件中读取角色→权限映射。

**鉴权流程（以私有空间为例）：**
```
请求 → @SaSpaceCheckPermission(value = "picture:upload")
  → StpKit.SPACE.checkPermission("picture:upload")
  → SpaceUserAuthManager.getPermissionList(space, loginUser)
    → 私有空间：判断是否为创建者或 admin
    → 团队空间：查 space_user 表获取角色，再映射权限
  → 权限不足则抛 NO_AUTH_ERROR
```

#### 5.3.3 权限配置（`spaceUserAuthConfig.json`）

| 角色 | 权限 |
|------|------|
| viewer（浏览者） | picture:view |
| editor（编辑者） | picture:view, picture:upload, picture:edit, picture:delete |
| admin（管理员） | spaceUser:manage, picture:view, picture:upload, picture:edit, picture:delete |

### 5.4 图片上传方案

采用**模板方法模式**，`PictureUploadTemplate` 为抽象基类，子类分别处理文件上传和 URL 上传。

```
PictureUploadTemplate（抽象类）
  ├── FilePictureUpload（处理 MultipartFile）
  └── UrlPictureUpload（处理 URL 字符串，下载后上传）
```

**上传流程：**
1. 校验登录状态及空间额度（仅空间场景）。
2. 根据入参类型（`MultipartFile` / `String URL`）选择对应上传策略。
3. 上传至 COS，获取图片元数据（URL、缩略图、大小、宽高、主色调、格式等）。
4. 填充审核参数（管理员/私有空间自动过审）。
5. 在**编程式事务**中保存图片记录并更新空间额度（`totalSize + picSize`，`totalCount + 1`）。
6. 返回图片 VO。

**COS 路径规则：**
- 公共图库：`public/{userId}/...`
- 私有空间：`space/{spaceId}/...`
- 头像：`Avatar/{userId}/...`

**图片删除：**
- 事务内删除数据库记录并释放空间额度。
- 事务外异步清理 COS 文件（若 URL 被多条记录共用则跳过）。

### 5.5 缓存方案

#### 双层缓存架构（图片列表查询）

```
请求 → 查 Caffeine 本地缓存（5 分钟 LRU）
       ↓（未命中）
      查 Redis 缓存（5~10 分钟随机过期，防雪崩）
       ↓（未命中）
      查 MySQL
       ↓
      写入 Redis → 写入 Caffeine → 返回结果
```

> 当前缓存接口标注了 `@Deprecated`，主接口直接查库，后续可根据流量情况开启缓存。

#### Redis 数据结构使用

| Key | 类型 | 用途 |
|-----|------|------|
| `search:hot:{searchType}` | ZSet | 热词统计，score = 搜索次数 |
| Spring Session Keys | Hash | 用户会话持久化 |
| Sa-Token Keys | String | Sa-Token 登录态 |
| `yunpicture:listPictureVOByPage:{md5}` | String | 图片列表查询缓存 |

### 5.6 分库分表方案

基于 ShardingSphere JDBC 5.2.0，对 `picture` 表按 `spaceId` 进行动态分表。

**分片算法（`PictureShardingAlgorithm`）：**
- `spaceId = null`（公共图库）：路由到主表 `picture`。
- `spaceId != null`：路由到 `picture_{spaceId}` 分表。
- 若分表不存在，降级路由到主表 `picture`。

**分表创建时机：** 创建团队/私有空间时，通过 `DynamicShardingManager` 动态建表（当前已注释，待补充）。

**适用场景：** 大型团队空间图片量大时，通过分表避免单表膨胀。

### 5.8 搜索历史与热词方案

**记录时机：** 用户在图片/用户/空间搜索时，若搜索文本不为空，触发异步记录（`CompletableFuture.runAsync`）。

**个人历史策略：**
- 同一用户同类型同关键词：先删后插（更新时间），避免重复。
- 每用户每类型最多保留 20 条，超过则删除最旧的。

**热词策略：**
- Redis ZSet：key = `search:hot:{searchType}`，每次搜索 `ZINCRBY +1`。
- 热词读取：`ZREVRANGE 0 9` 取 Top 10。
- 未登录用户也会更新热词，但不记录个人历史。

## 7. 关键流程说明

### 7.1 公共图库图片发布流程

```
普通用户上传图片
  → reviewStatus = 0（待审核）
  → 管理员调用 /picture/review（通过/拒绝）
  → reviewStatus = 1（通过）→ 图片在公共图库可见
  → reviewStatus = 2（拒绝）→ 图片不可见

管理员/私有空间上传图片
  → reviewStatus = 1（自动过审）→ 直接可见
```

### 7.2 团队空间成员入驻流程

```
方式一（管理员直接添加）：
  管理员 POST /spaceUser/add → status=1（已通过）→ 成员立即有权限

方式二（用户申请加入）：
  用户 POST /spaceUser/apply → status=0（待审批）
  管理员 POST /spaceUser/approve → status=1（通过）或 status=2（拒绝）
  通过后 → 成员具备对应角色权限
```

### 7.3 AI 扩图流程

```
用户点击扩图
  → POST /picture/out_painting/create_task（提交图片 URL + 扩图参数）
  → 返回 taskId
  → 前端轮询 GET /picture/out_painting/task?taskId=xxx
  → 状态 SUCCEEDED → 获取结果图片 URL
  → 用户选择保存 → POST /picture/upload/url（将结果图保存到空间）
```

### 7.4 点赞事务流程

```
用户点赞图片
  → 查询 picture_like 表是否已有记录
  → 已有 → 删除记录 + picture.likeCount - 1（取消点赞）
  → 没有 → 插入记录 + picture.likeCount + 1（点赞）
  → 以上操作在同一事务（@Transactional）中完成
  → 返回 LikeVO（isLiked, likeCount）
```

### 7.5 搜索记录异步写入流程

```
用户提交图片搜索（含关键词）
  → 主链路：查 MySQL 返回图片列表
  → 异步（CompletableFuture）：
      ① ZINCRBY Redis ZSet 热词 +1
      ② 已登录：去重后写入 search_history 表
      ③ 超过 20 条历史 → 删除最旧记录
```
